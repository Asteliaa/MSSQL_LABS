# План выполнения лабораторных работ MSSQL Lab

Документ задает порядок выполнения лабораторных работ, правила оформления артефактов и критерии приемки в формате Ubuntu + Docker + sqlcmd.

## Принципы выполнения

1. Сначала настраивается и проверяется окружение, затем выполняются лабораторные по порядку.
2. Каждая лабораторная хранит команды, скрипты, отчет и скриншоты отдельно.
3. Любой шаг в отчете должен быть воспроизводим по файлам из `commands/` и `scripts/`.
4. Исходные требования, где упоминаются Hyper-V/SSMS, адаптируются к Docker/sqlcmd и явно фиксируются в README лабораторной.

## Фазы проекта

1. Подготовка окружения.
2. ЛР1: установка и проверка двух экземпляров SQL Server в Docker.
3. ЛР2: базы данных, файлы и файловые группы.
4. ЛР3: резервное копирование и восстановление.
5. ЛР4: безопасность.
6. ЛР5: автоматизация задач.
7. ЛР6: репликация и HA.
8. ЛР7: мониторинг и диагностика.
9. Финальная проверка целостности репозитория.

## Обязательные артефакты для каждой лабораторной

В каждой директории `labs/XX-.../` должны быть:

- `README.md` с описанием цели и адаптации задания под Docker.
- `commands/` с реальными выполненными командами.
- `scripts/` с исполняемыми SQL или shell скриптами.
- `report/report.md` с ходом выполнения и результатами.
- `screenshots/` с подтверждающими изображениями.

## Шаблон отчета

Рекомендуемая структура `report/report.md`:

1. Тема и цель лабораторной.
2. Исходное задание и Docker-адаптация.
3. Используемое окружение.
4. Пошаговое выполнение.
5. Проверочные запросы и результаты.
6. Вывод и соответствие требованиям.

## Критерии приемки по лабораторным

### ЛР1. Установка SQL Server

Папка: `labs/01-installation/`

Критерии Done:

1. Запущены контейнеры `mssql-default` и `mssql-named`.
2. Подключение к каждому экземпляру подтверждено запросом `SELECT @@VERSION`.
3. Показаны свойства экземпляров и системные БД.
4. Проверены остановка и запуск экземпляра через `docker compose stop/start`.
5. Второй экземпляр доступен через отдельный порт на хосте.

### ЛР2. Управление БД и файлами

Папка: `labs/02-databases-and-files/`

Критерии Done:

1. В default экземпляре создана БД `Test` с параметрами файла `testdata_a` (4MB, growth 2MB, max 10MB).
2. Создана файловая группа `TestFileGroup`.
3. Добавлен файл `testdata_b.ndf` (5MB, growth 2MB, unlimited) в `TestFileGroup`.
4. В named экземпляре создана БД с инициалами, например `RZ_DB`.
5. Созданы схемы и таблицы, включая таблицу на явной файловой группе.
6. Результаты подтверждены запросами к `sys.database_files`, `sys.filegroups`, `sys.tables`, `sys.schemas`.

### ЛР3. Резервное копирование и восстановление

Папка: `labs/03-backup-and-recovery/`

Критерии Done:

1. Выполнен полный backup выбранной БД в `docker/backups/`.
2. Выполнен минимум один дополнительный тип backup по заданию (diff или log).
3. Показана проверка backup-файлов и метаданных.
4. Отработан сценарий восстановления на тестовой БД.
5. В отчете есть проверка консистентности данных после restore.

### ЛР4. Безопасность

Папка: `labs/04-security/`

Критерии Done:

1. Созданы логины и пользователи на уровне сервера и БД.
2. Назначены роли и явные права на объекты.
3. Проверено, какие действия доступны и недоступны тестовому пользователю.
4. В отчете есть матрица прав до и после изменений.

### ЛР5. Автоматизация

Папка: `labs/05-automation/`

Критерии Done:

1. Реализован повторяемый сценарий автоматизации (скрипт или job-подобный workflow).
2. Показан запуск по расписанию или имитация расписания.
3. Реализовано логирование результата выполнения.
4. Есть проверка успешного и ошибочного сценария.

### ЛР6. Репликация и HA

Папка: `labs/06-replication-ha/`

Критерии Done:

1. Описана выбранная модель отказоустойчивости или репликации в рамках Docker-стенда.
2. Показаны шаги настройки и ограничения учебного сценария.
3. Подтверждена доступность данных после имитации отказа или переключения.
4. В отчете есть вывод о применимости схемы в production и ее ограничениях.

### ЛР7. Мониторинг и диагностика

Папка: `labs/07-monitoring/`

Критерии Done:

1. Собраны базовые метрики SQL Server через DMV.
2. Показан поиск проблемного запроса или узкого места.
3. Выполнено улучшение и повторное измерение.
4. В отчете зафиксированы исходные и итоговые показатели.

## Правила качества артефактов

1. Команды в `commands/` должны совпадать с фактически выполненными.
2. Скрипты в `scripts/` должны быть запускаемыми без ручного редактирования по месту.
3. Имена файлов в отчете и `screenshots/` должны совпадать по расширению и написанию.
4. В отчете нельзя оставлять черновые пометки и неразрешенные ссылки.
5. Каждая лабораторная должна иметь явный блок проверки результата.

## Финальная проверка проекта

Перед сдачей проекта убедись, что:

1. Запуск окружения описан единообразно в README и docs.
2. Для ЛР1 и ЛР2 есть полный набор артефактов и доказательств.
3. Для ЛР3-ЛР7 заполнены минимум README, commands, scripts и report.
4. Все команды и скрипты воспроизводимы на Ubuntu без дополнительных скрытых шагов.
