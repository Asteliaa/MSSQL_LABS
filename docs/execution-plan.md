# План выполнения лабораторных работ по администрированию Microsoft SQL Server в Docker

Этот документ фиксирует порядок выполнения лабораторных работ, структуру артефактов по каждой работе и правила ведения проекта. Он нужен для того, чтобы сначала настраивалось единое окружение, а уже затем последовательно выполнялись лабораторные задания в воспроизводимом виде через Docker, VS Code и Git.

## Общая логика работы

Выполнение всех лабораторных работ делится на два больших этапа:

1. Подготовка окружения и инфраструктуры проекта.
2. Последовательное выполнение лабораторных работ по темам курса.

Такой порядок позволяет не смешивать системную настройку с содержанием самих лабораторных и соответствует удобной практике ведения репозиториев для учебной и лабораторной деятельности.

## Этап 1. Подготовка окружения

Перед выполнением лабораторных работ необходимо:

- установить Docker и проверить работу `docker compose`.
- установить VS Code и расширения для Docker, Markdown, Git и SQL.
- установить Git и настроить имя пользователя и email.
- создать структуру проекта и инициализировать Git-репозиторий.
- подготовить файлы `docs/environment-setup.md` и `docs/execution-plan.md` как базовую документацию проекта.

## Этап 2. Выполнение лабораторных работ

Лабораторные работы выполняются по порядку, каждая в своей папке внутри `labs/`. Для каждой работы должны храниться как минимум отчёт, журнал команд, скрипты и скриншоты. Такой подход упрощает навигацию по проекту и отделяет оформленную часть работы от технических артефактов выполнения.

### Порядок лабораторных работ

| Лабораторная | Папка проекта | Содержание |
|---|---|---|
| ЛР 1. Установка SQL Server | `labs/01-installation/` | Развёртывание окружения SQL Server в Docker, подключение, проверка экземпляров, порты |
| ЛР 2. Управление БД и файлами | `labs/02-databases-and-files/` | Создание БД, работа с файлами данных и журналов |
| ЛР 3. Резервное копирование и восстановление | `labs/03-backup-and-recovery/` | Бэкапы, восстановление, аварийные сценарии |
| ЛР 4. Управление безопасностью | `labs/04-security/` | Логины, пользователи, роли и права доступа |
| ЛР 5. Автоматизация задач | `labs/05-automation/` | Джобы, SQL Server Agent, maintenance tasks |
| ЛР 6. Репликация и высокая доступность | `labs/06-replication-ha/` | Репликация, порты, доступность, варианты HA |
| ЛР 7. Мониторинг и поиск неисправностей | `labs/07-monitoring/` | Activity Monitor, DMVs, диагностика запросов |

Тематика этих работ соответствует предоставленным учебным материалам по установке, управлению базами, безопасности, автоматизации, HA и мониторингу SQL Server.

## Обязательная структура внутри каждой лабораторной

Для каждой лабораторной работы используется единый шаблон:

- `README.md` — краткое описание лабораторной, цели и содержимого папки.
- `report/report.md` — оформленный отчёт по выполнению лабораторной работы.
- `commands/` — журнал всех выполненных команд и шагов.
- `scripts/` — SQL-, shell- и другие исполняемые файлы.
- `screenshots/` — изображения для подтверждения выполнения шагов.

Единая структура внутри всех лабораторных делает проект предсказуемым и удобным для проверки, а также хорошо подходит для контроля истории изменений через Git.

## Правила заполнения лабораторных папок

### README.md

В `README.md` каждой лабораторной кратко указывается:

- название лабораторной работы;
- цель;
- краткий перечень файлов;
- итоговый статус выполнения.

### report/report.md

В отчёте рекомендуется соблюдать структуру:

1. Тема лабораторной работы.
2. Цель работы.
3. Используемое окружение.
4. Ход выполнения.
5. Результаты.
6. Вывод.

### commands/

В папке `commands/` нужно хранить реальные команды, которые были выполнены в терминале или SQL-клиенте. Лучше разбивать их на несколько файлов по смыслу, например:

- `01-docker-setup.md`
- `02-connectivity-check.md`
- `03-validation.sql`

Такой подход удобен для проверки и повторного запуска отдельных частей лабораторной.

### scripts/

В `scripts/` помещаются исполняемые скрипты:

- `.sql`
- `.sh`
- `.ps1`

Скрипты полезно хранить отдельно от «журнальных» файлов, чтобы их можно было переиспользовать без ручного копирования из Markdown.

### screenshots/

В папке `screenshots/` хранятся все скриншоты, которые подтверждают выполнение шагов лабораторной работы. Имена файлов лучше делать понятными, например:

- `01-docker-ps.png`
- `02-ssms-connection.png`
- `03-query-result.png`
